1 # LuaUtils # {#lua_utils}
2 [TOC]
3 
4 LuaUtils is a component that extends the Lua environment with some commonly used functionality.
5 
6 It can be accessed from Lua scripts via the 'luautils' global variable. This component mostly does a lot of setup work in the background, but it also provides some handy functions for instantiating, finding and destroying gameobjects in the scene.
7 
8 # Example # {#lua_utils_example}
9 
10 Here's an example of the kind of thing you can do:
11 
12 ```lua
13 local go = luautils.Find("MyObject") -- Find a game object by name
14 luautils.Destroy(go) -- Destroy it
15 ```

43 # Registering C# Types # {#registering_types}
44 
45 The most important function of the LuaUtils component is registering C# types so that instances of those types can be accessed from Lua scripts.
46 
47 In order to access the members of a C# type from Lua, the type first has to be registered with MoonSharp. Note that for objects added using the LuaBindings component, the relevant types are registered automatically.
48 
49 In some cases however, you will need to register a type explicitly. The easiest way to do this is by adding the type's name to the FungusTypes.txt or UnityTypes.txt JSON files referenced by the LuaUtils component. You can also create your own JSON files to register additional types. Note that types that are not contained in the main application DLL will need to use the [namespace qualified type name] in the JSON file.
50 
51 # Example JSON Type File # {#example_json_file}
52 
53 Example of a types JSON file:
54 ```json

65 # Registering Types Directly # {#register_types_directly}
66 
67 If you need to register types directly from C#, or do a more complex type of registration, you can use the MoonSharp UserData class to do this. See the MoonSharp documentation for a list of supported registration methods. A good place to register C# types is in the Awake method of a custom component.

84 # PODFactory # {#pod_factory}
85 
86 Due to limitations in C# / Mono, MoonSharp has limited support for working with Plain-Old-Data (struct) types like [Vector3](http://docs.unity3d.com/ScriptReference/Vector3.html), [Color](http://docs.unity3d.com/ScriptReference/Color.html), etc.
87 
88 The best approach here is to treat POD properties as immutable objects, and never try to modify a POD variable that has been acquired from a C# object. Instead, you should construct a new POD object, populate it with the required values and then pass that object in calls to C# code. The LuaUtils PODFactory class helps do this for common Unity types.
89 
90 ```lua
91 -- Returns a new Color object
92 local c = luautils.factory.color(1,1,1,1)
93 
94 -- Returns a new Vector2 object
95 local v2 = luautils.factory.vector2(1, 2)
96 
97 -- Returns a new Vector3 object
98 local v3 = luautils.factory.vector3(1, 2, 3)
99 
100 -- Returns a new Vector4 object
101 local v4 = luautils.factory.vector4(1, 2, 3, 4)
102 
103 -- Returns a new Quaternion object
104 local q = luautils.factory.quaternion(float x, float y, float z) -- Rotation in euler angles
105 
106 -- Returns a new Rect object
107 local r = luautils.factory.rect(float x, float y, float width, float height)
108 ```
