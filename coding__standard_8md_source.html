1 # Coding Standard # {#coding_standard}
2 
3 \brief The %Fungus coding standard is designed to make the source code simple to understand for users and easy to maintain for contributors.
4 
5 This document is focussed on decisions that we've made for the %Fungus project. For general Unity coding tips, try [50 Tips and Best Practices for Unity](http://www.gamasutra.com/blogs/HermanTulleken/20160812/279100/50_Tips_and_Best_Practices_for_Unity_2016_Edition.php).

7 # Code layout # {#code_layout}
8 
9 This is the typical layout of a class in %Fungus:
10 
11 ```
12 using UnityEngine;
13 
14 namespace Fungus
15 {
16  /// <summary>
17  /// Description of the component.
18  /// </summary>
19  public class MyComponent : MonoBehaviour
20  {
21  [Tooltip("Tooltip comment displayed in the editor.")]
22  [SerializeField] protected float myProperty = 10f;
23 
24  protected virtual void MyMethod()
25  {
26  if (myProperty > 5f)
27  {
28  Debug.Log("A message");
29  }
30  }
31 
32  #region Public members
33 
34  /// <summary>
35  /// Documentation comment for public property.
36  /// </summary>
37  public virtual float MyProperty { get; set; }
38 
39  /// <summary>
40  /// Aspect ratio of the secondary view rectangle. e.g. a 2:1 aspect ratio = 2/1 = 2.0.
41  /// </summary>
42  public virtual void DoSomething()
43  {
44  }
45 
46  #endregion
47  }
48 }
49 ```
50 
51 Things to note:
52 
53 - using declarations all go together at the top of the file.
54 - You should remove any unused using declarations (can spot these easily with static code analysis - see below).
55 - Runtime code goes in the Fungus namespace.
56 - Editor code goes in the Fungus.EditorUtils namespace.
57 - All public classes, structs, enums and class members should be documented using xml comments.
58 - You can document private and protected members if you want to, but ALL public members must have at least a summary comment.
59 - Parameter & return descriptions are optional, add them if you feel the parameters require a non-trivial explanation.
60 - Serialized fields should NEVER be public. Use a public accessor property to access the field if external access is required.
61 - All serialized fields should have a Tooltip attribute. This doubles as code documentation for the field.
62 - All methods should be declared virtual and use protected instead of private. This allows for easy inheritance and extension (at the cost of some performance).
63 - All public members of a class (including public static & delegate types) should be placed inside a 'Public members' region for easy access.

65 # Coding best practices # {#coding_best_practices}
66 
67 These are some general best practices when writing code for %Fungus. Where these go against the usual recommended coding practice (e.g. Assert) it's because of an issue in Unity with doing it 'the right way'.
68 
69 - Use the static code analyser in MonoDevelop. http://tinyurl.com/h7xqpwg
70 - Use the c# xml comment style. https://msdn.microsoft.com/en-us/library/b2s063f7.aspx
71 - Declare all public enums at namespace scope, not inside a class. (Consistency, easier sharing of enums between classes).
72 - Use var instead of declaring variable types when possible. (More readable).
73 - Use for instead of foreach when possible. (Avoids allocating an iterator & GC problems).
74 - Use string.Format or StringBuilder instead of concatenating strings. (Avoids allocations & GC problems).
75 - Don't use LINQ. (Avoids allocations & GC problems.)
76 - Don't use Assert. (We support back to Unity 5.0, before Assert was introduced).
77 - Use Mathf.Approximately when comparing float variables to constants.
78 - Treat compiler warnings as errors. There should be zero warnings at build or runtime in normal operation.
79 - Add global constants to FungusConstants.cs
80 - Always try to maintain backwards compatibility when introducing changes. We support Unity 5.0+ so beware of API differences in newer versions.
